01 · Goal

目标与边界

回答四件事：为什么要做 AI-native biotech、第一年验证什么、哪些资产会形成长期复利，以及第一阶段不做什么。

Company

长期目标

建设 AI-native biotech：连接科学知识、蛋白设计模型、专家判断、实验反馈和肿瘤治疗场景，让 AI 不只是工具，而是研发流程的组织者。

  • AI-native biotech 目标：AI 参与靶点判断、蛋白质候选生成、实验设计、CRO / 伙伴调度、结果复盘和资产决策，而不是停留在报告生成或 API 调用。
  • 科学主线目标：蛋白质设计 + 肿瘤治疗；器官芯片 / 类器官不是另一个散点方向，而是为肿瘤治疗判断提供 phenotype reward。
  • 产业目标：把外部合作项目逐步沉淀为可判断、可优化、可对外合作的资产：候选蛋白、实验数据、项目证据、BD 材料和后续共研 / 分成机会。
Year 1

第一年验证

不在页面提前锁死第一单 target，而是用 1 个外部伙伴共创的肿瘤蛋白设计项目 + 1 个关键 ML 任务验证四件事：AI4AI 平台底座、蛋白质设计能力、肿瘤 reward 闭环、产业价值。

  • AI4AI 平台验证：Scientific Memory Graph、Scientific Tool Orchestration、Review & Reward Ledger、Training Data Compiler 和 agent 平台本身完成 v0 验收，能支撑真实项目执行、回放和数据编译
  • 蛋白质设计验证：围绕外部伙伴确认的肿瘤 target 或蛋白治疗问题，完成 binder / antibody / nanobody / protein candidate 的生成、筛选和排序
  • 肿瘤 reward 验证：第一层验收 expression、binding、developability；第二层用肿瘤类器官 / 器官芯片做小规模 phenotype reward pilot
  • 产业验证：为 1 个项目型 biotech 完成 Target-to-Binder Sprint，8-12 周内形成候选排序、实验反馈和继续 / 停止 / BD / 下一轮实验建议
Asset

复利资产

这些资产服务 AI-native biotech，而不只是平台交付。Agent 平台负责把项目跑起来，Memory 是 source of truth，Tool、Ledger 和 Compiler 分别负责工具调用、科学判断和训练自进化。

  • Agent Runtime & Project Room：项目空间、权限、任务计划、审批 gate、执行日志和交付导出
  • Scientific Memory Graph：文献、数据库、claim-evidence-provenance、项目状态、版本和冲突证据
  • Scientific Tool Orchestration：深度学习模型、计算工具、数据库 API、MCP 服务、wet-lab 接口
  • Review & Reward Ledger：expert taste、rubric、wet reward、human override、失败原因和 QC flags
  • Training Data Compiler：基于内部轨迹、schema、rubric、匿名统计和授权数据形成 SFT、preference、critique、failure recovery 和 agent self-evolve dataset
Boundary

第一阶段边界

第一阶段不是卖通用科研助手、总部级平台订阅，也不是立刻自建完整临床管线；而是用 Target-to-Binder Sprint 服务有明确决策窗口的肿瘤 biologics discovery team。

  • 客户与伙伴边界：首批合作对象锁定肿瘤 biologics / protein therapeutics 早研团队；ADC、双抗、TCE 是 binder 的下游转化形态，不是并列的第一阶段产品线。
  • 产品边界：Target-to-Binder Sprint 是第一单的最小可交付工作包；Project Room 是交付载体，也是 AI-native biotech 研发系统的产品雏形。
  • 市场边界：第一阶段只服务有明确靶点、疾病场景或候选分子的肿瘤蛋白药早研项目，帮助客户在 8-12 周内决定继续投入、停止、对外 BD，或进入下一轮实验；不覆盖小分子全流程、临床开发、常规 CRO 外包和通用科研工具。
  • 数据边界：客户原始数据、项目材料、序列和实验结果默认不用于训练；schema、rubric、workflow、匿名统计、授权数据和内部生成轨迹可用于平台改进。

02 · Users

目标用户

第一批合作对象优先级已经确定：项目型肿瘤 biotech 负责真实项目牵引，大药企负责标杆试点，科研机构负责高质量项目源，wet-lab / organ-chip 平台负责 reward 壁垒。

产业客户：直接付费

首批直接付费客户锁定肿瘤 biologics / protein therapeutics 项目型 biotech。典型规模 20-200 人，已有 target、disease hypothesis 或 candidate，需要 8-12 周内完成 continue / kill / BD / next experiment 判断。

03 · Sci Mainline

Sci 主线与项目入口

具体做什么单子，需要后续和相关生物公司谈；页面要先讲清楚我们自己的 Sci 主线：蛋白质设计产生候选，肿瘤实验模型给 reward，AGI4Sci 平台把每一轮设计、验证和学习沉淀下来。

Mainline A

蛋白质设计引擎

从肿瘤 target、疾病假设或已有候选出发，生成和优化 binder、antibody、nanobody 或其他 protein candidate。

  • 解决的问题：外部伙伴知道肿瘤方向和靶点压力，但缺少快速、可复盘的候选生成、排序和失败解释流程。
  • 第一阶段交付：候选序列 / 结构、靶点证据、epitope / specificity / developability 风险标签、Top candidates 和下一轮实验建议。
Mainline B

肿瘤 reward 系统

expression、binding、developability 是第一层 wet reward；肿瘤类器官 / 器官芯片提供更接近治疗场景的 phenotype reward。

  • 解决的问题：只看 binding 容易停在分子层，无法判断候选是否有肿瘤组织层面的有效性、穿透、选择性或毒性风险。
  • 第一阶段交付：标准化实验任务书、数据回传模板、reward ledger、negative samples、失败原因和可用于训练的数据格式。
First Deal

第一单由伙伴共创

不提前在内部页面拍死某个 target 或公司，先定义项目选择标准，再和相关生物公司谈具体单子。

  • 选择标准：肿瘤 biologics 项目、有明确 target / candidate、有 8-12 周决策压力、能做 expression / binding / developability，且可接入小规模肿瘤模型 reward。
  • 商务形态：可以是付费 Sprint、共研项目、标杆试点或带后续权益的合作，但第一阶段必须沉淀平台数据和决策流程。
Therapy Path

下游治疗形态选择

ADC、双抗、TCE 和其他 biologics 不是第一阶段互相割裂的方向，而是 binder / protein candidate 的不同转化路径。

  • 解决的问题：同一个 target 的候选，后续到底适合 ADC、双抗、TCE、单抗、融合蛋白或其他蛋白治疗，需要由证据和 reward 共同决定。
  • 第一阶段交付：候选排序 + 治疗路径建议 + 下一轮实验设计，而不是在项目开始前先把所有形态都当成并列产品线。

05 · Delivery System

项目交付系统

这一层回答 Target-to-Binder Sprint 如何被生产出来：从项目输入，到工作区推进，再到交付档案、实验数据包和肿瘤 reward，最后在授权边界内沉淀成下一次可复用的研发流程。

项目输入

客户给出 target、肿瘤疾病假设、候选分子、项目约束、成功标准和停止标准。第一步不是跑模型，而是确认项目是否适合蛋白质设计、能否做标准 wet reward，并把数据默认不训练、授权复用边界和判断口径说清楚。

项目工作区

所有证据、候选、实验、专家意见和决策记录进入同一个 Web 工作区。客户看到的是项目进展和判断依据，不需要理解底层技术模块。

交付生成

按用户场景导出候选物决策档案、论文材料、实验与数据包、管理层摘要或内部试点评估材料，让项目结果能被继续使用。

项目沉淀

每个项目结束后，在授权范围内留下模板、数据标准、失败案例、评审规则、实验反馈和复盘记录，成为下一次项目更快、更准的基础。

06 · Technical Architecture

平台技术架构

这一层是衔远内部要沉淀的 AI4AI / AGI4Sci 平台能力：底层复用 AI4AI 方法，产品形态面向 AI-native biotech，把 agent 平台、科研记忆、蛋白设计工具、专家判断、肿瘤 reward 和可回放轨迹组织成持续改进的系统。

1

2

07 · Vertical Loop Example

第一条 Sci 闭环：Protein Design to Tumor Reward

这不是要在内部先拍死某个具体单子，而是把所有后续合作都约束到同一条主线：蛋白质设计产生候选，标准 wet assay 给第一层 reward，肿瘤类器官 / 器官芯片给 phenotype reward。

一次项目怎么跑

从客户输入到决策交付，每一步都留下证据、版本和复盘材料。

1客户和外部伙伴输入肿瘤 target、疾病场景、候选物或项目约束，明确成功标准和停止标准
2衔远系统整理文献、数据库、竞品、结构信息和项目材料，形成可追溯靶点证据图
3衔远 AGI4Sci 平台调度模型和工具，生成或筛选 binder、antibody、nanobody 或 protein candidate，并给出风险标签
4外部专家评审候选排序、机制假设、成药性风险和实验可行性，评审过程进入 rubric
5Top 20-30 candidates 进入外部 wet-lab partner 的 expression、binding、developability 小规模验证
6Top 3-5 candidates 进入肿瘤类器官 / 器官芯片 phenotype reward pilot，观察更接近治疗场景的信号
7输出继续、停止、BD、补实验、下一轮优化或 ADC / 双抗 / TCE 等治疗路径建议
8把失败案例、实验反馈、专家判断和工具轨迹回流到衔远 AGI4Sci 平台

为什么先从这里开始

它同时具备科学主线、商业清晰度、实验反馈和长期资产化价值。

1主线清楚：蛋白质设计是生成能力，肿瘤 reward 是学习信号，二者合起来才是 AI-native biotech
2关系清楚：Target-to-Binder 是前端工作包，ADC / 双抗 / TCE 是候选蛋白后续可能进入的治疗形态
3周期可控：8-12 周能形成候选排序、标准 wet reward 和下一轮实验判断
4反馈真实：器官芯片 / 类器官把项目从单纯 binding 推向肿瘤治疗场景，但第一年先做小规模 reward pilot
5商业灵活：具体第一单后续和生物公司谈，可以是付费 Sprint、共研项目或带后续权益的合作

08 · Validation Plan

18 个月验证计划

每个阶段同时验收三条线：外部伙伴能否带来真实肿瘤项目；蛋白质设计 + 肿瘤 reward 闭环是否成立；衔远 AI4AI 平台的 agent runtime、Memory Graph、Tool Orchestration、Review & Reward Ledger、Training Data Compiler 是否能在真实项目中完成可验收的搭建、运行和复用。

0-3 个月

AI-native biotech 底座与项目共创标准

不先拍死 target，而是确定第一单选择标准：肿瘤 biologics 项目、蛋白质设计可介入、标准 wet reward 可做、肿瘤类器官 / 器官芯片 reward 有接入可能；衔远先把可运行底座搭起来。

  • 外部项目：确定候选生物公司、target / disease hypothesis 选择标准、成功标准、停止标准、专家 rubric、实验伙伴 shortlist 和数据字段。
  • Agent 平台：项目空间、租户隔离、权限审计、任务模板、human approval gate 和运行日志上线。
  • Memory Graph：claim-evidence-provenance schema、target / candidate / assay / entity model、文献 / 专利 / 数据库 ingest v0。
  • Tool Orchestration：文献检索、结构 / 序列工具、蛋白候选生成、数据库 API 的 tool registry 和标准 I/O。
  • Review & Reward Ledger：专家 rubric、候选评分、人工 override、QC flags、wet reward 和 tumor phenotype reward 字段定义。
  • Training Data Compiler：trace schema、脱敏 / 授权边界、SFT / preference / critique / failure-recovery 数据格式定义。
3-6 个月

蛋白质设计 dry-run 与 reward 接口

围绕伙伴共创项目完成 dry-run：证据整理、蛋白候选生成、专家评审和候选排序；同步把标准 wet assay 与肿瘤 reward 接口接入项目系统。

  • 外部项目：生成 50-300 条候选，Top 20-30 条完成外部专家评审和 expression / binding / developability 优先级排序。
  • Agent 平台：能自动生成任务计划、预算 / 时间记录、失败重试、审批节点和项目时间线。
  • Memory Graph：每条关键结论能追溯来源、冲突证据、版本和引用。
  • Tool Orchestration：50-300 条蛋白候选的生成、筛选、去重和风险标注可复现，工具输入输出可回放。
  • Review & Reward Ledger：Top 20-30 条候选完成专家评审，评分差异、反驳理由和实验优先级结构化入库。
  • Training Data Compiler：dry-run 轨迹可导出为训练 / 评测样本，包含成功、失败、人工修正和恢复路径。
6-12 个月

wet reward 与肿瘤模型 pilot

Top 20-30 candidates 进入外部伙伴的表达、结合和成药性验证；Top 3-5 candidates 进入肿瘤类器官 / 器官芯片 phenotype reward pilot；实验回流后检查 AI4AI 数据飞轮是否成立。

  • 外部项目：交付候选物决策档案，给出继续、停止、BD、补实验或治疗路径建议。
  • Agent 平台：wet-lab SOP、样本表、状态、raw data、QC flags 接入项目流。
  • Memory Graph：实验结果回写到 target、candidate、evidence、assay 和 tumor model 节点，形成版本化项目记忆。
  • Tool Orchestration：能根据 wet feedback 触发候选重排、补实验建议或下一轮候选生成。
  • Review & Reward Ledger：raw data、negative samples、失败原因、专家复评、wet reward 和 tumor phenotype reward 全量可查询。
  • Training Data Compiler：形成首批 SFT / preference / critique / failure-recovery / eval set，不含客户未授权原始数据。
  • AI4AI 方法：关键 ML benchmark 达到强基线，并沉淀 harness、评测流程和失败复盘模板。
12-18 个月

第二项目复用与资产化判断

判断同一套 AGI4Sci 平台能否接入第二个外部伙伴项目，并让启动成本、交付周期和人工组织成本明显下降；同时判断是否从服务项目升级到共研权益、里程碑或分成。

  • 外部项目：第二个外部项目能复用 target intake、protein design workflow、expert rubric、实验数据模板和决策档案结构。
  • Agent 平台：复用项目模板、权限模型、任务计划、审批 gate 和交付导出。
  • Memory Graph：复用 schema 和 ingest pipeline，新增 target 的初始化时间下降。
  • Tool Orchestration：复用工具链、评测 harness 和候选生成流程，关键步骤无需重写。
  • Review & Reward Ledger：复用 rubric / wet reward schema，能跨项目比较成功和失败模式。
  • Training Data Compiler：形成可持续训练 / 评测数据生产线，支持下一轮 agent 能力迭代和私有部署判断。
  • 经营判断：决定是否继续做付费 Sprint、垂直 project room、私有部署、共研权益或资产分成，而不是只做工具/API 供应商。
