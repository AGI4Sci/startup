01 · Goal

目标与边界

回答四件事：第一阶段先建设哪些 AI4AI 核心能力，Sci 项目如何牵引这些能力增长，哪些资产会形成长期复利，以及第一阶段不做什么。

Company

长期目标

建设可迁移的 AI4AI 核心能力：让 agent 能在高复杂度专业任务中计划、执行、调用工具、记忆证据、接受反馈、生成训练数据，并持续改进。

  • AI4AI 目标：沉淀 agent runtime、memory、tool orchestration、review / reward、training data compiler 和 eval / replay harness，形成可跨领域复用的底座。
  • AGI4Sci 关系：Sci 不是第一天并列的新业务线，而是 AI4AI 的第一批真实高难度任务场；蛋白质设计和肿瘤 reward 提供真实工具调用、专家判断和 wet feedback。
  • 长期 Sci 选项：当 AI4AI × Sci 闭环被 2-3 个真实项目验证后，再判断是否进入 AI-native biotech、共研权益、里程碑或资产分成。
Year 1

第一年验证

第一年不把具体 Sci 项目放在 AI4AI 前面，而是分两条线推进：内部先把 AI4AI v0 做成，外部同步建立 Sci 合作项目池；等底座可运行后，用 1 个真实 Sci 项目牵引能力增长。

  • AI4AI 平台验证：Scientific Memory Graph、Scientific Tool Orchestration、Review & Reward Ledger、Training Data Compiler 和 agent runtime 完成 v0 验收，能支撑复杂任务执行、回放、评测和数据编译。
  • Sci 合作验证：联系蛋白质设计、肿瘤 biologics、类器官 / 器官芯片相关伙伴，形成候选项目池、实验接口和数据授权边界。
  • 真实项目验证：在底座可运行后选择 1 个外部伙伴共创项目，围绕肿瘤 target 或蛋白治疗问题完成候选生成、排序、专家评审和小规模 wet reward。
  • 增长验证：把真实项目产生的证据、失败样本、专家判断、实验反馈和执行轨迹编译成 eval set / training data，用于下一轮 AI4AI 能力迭代。
Asset

复利资产

这些资产首先服务 AI4AI 核心增长，而不是只服务某个 Sci 单子。Agent 平台负责把任务跑起来，Memory 是 source of truth，Tool、Ledger 和 Compiler 分别负责工具调用、反馈记录和训练自进化。

  • Agent Runtime & Project Room：项目空间、权限、任务计划、审批 gate、执行日志和交付导出
  • Scientific Memory Graph：文献、数据库、claim-evidence-provenance、项目状态、版本和冲突证据
  • Scientific Tool Orchestration：深度学习模型、计算工具、数据库 API、MCP 服务、wet-lab 接口
  • Review & Reward Ledger：expert taste、rubric、wet reward、human override、失败原因和 QC flags
  • Training Data Compiler：基于内部轨迹、schema、rubric、匿名统计和授权数据形成 SFT、preference、critique、failure recovery 和 agent self-evolve dataset
Boundary

第一阶段边界

第一阶段不是立刻启动完整 biotech、不是重资产自建临床管线，也不是先追求多个 Sci 收入单；重点是把 AI4AI 核心能力做扎实，同时把 Sci 真实项目接进来作为牵引。

  • 优先级边界：内部执行的第一 KPI 是 AI4AI core 是否可运行、可评测、可训练；Sci 合作是牵引和验证，不替代核心能力建设。
  • Sci 边界：首批合作对象锁定肿瘤 biologics / protein therapeutics 早研团队；ADC、双抗、TCE 是候选蛋白的下游转化形态，不是并列的第一阶段产品线。
  • 产品边界：Target-to-Binder Sprint 只是 Sci 项目成熟后的最小可交付工作包；Project Room 是交付载体，也是 AI4AI agent workflow 的可视化验证场。
  • 市场边界：第一阶段不覆盖小分子全流程、临床开发、常规 CRO 外包和通用科研工具；也不承诺第一年形成完整 biotech 管线。
  • 数据边界：客户原始数据、项目材料、序列和实验结果默认不用于训练；schema、rubric、workflow、匿名统计、授权数据和内部生成轨迹可用于平台改进。

03 · Progression

AI4AI 与 AGI4Sci 的递进关系

具体做什么 Sci 单子，需要后续和生物公司谈；现在要先把递进关系讲清楚：先建 AI4AI 核心底座，再用 Sci 真实项目检验它，最后把项目中的证据、失败、reward 和轨迹编译回 AI4AI。

Step 1

AI4AI 核心能力先行

先把 agent runtime、memory graph、tool orchestration、review ledger、training data compiler 和 eval / replay harness 做成可运行系统。

  • 解决的问题：如果没有可复用底座，Sci 项目只会变成一次性咨询或外包交付，无法带来 AI4AI 增长。
  • 第一阶段产物：任务计划、证据记忆、工具注册、审批 gate、执行回放、评测样本和训练数据格式。
Step 2

Sci 合作方并行建立

在 AI4AI v0 建设期，同步联系蛋白质设计、肿瘤 biologics、类器官 / 器官芯片和 wet-lab partner，形成真实项目池。

  • 解决的问题：AI4AI 不能只在内部 benchmark 里自转，需要真实专家、真实实验约束和真实失败样本。
  • 第一阶段产物：候选合作方、项目选择标准、数据授权边界、实验接口、专家 rubric 和 reward 字段。
Step 3

真实 Sci 项目作为压力测试

当 AI4AI 底座可运行后，选择 1 个外部伙伴共创项目，以蛋白质设计和肿瘤 reward 为优先方向，让系统在真实任务中暴露问题。

  • 选择标准：肿瘤 biologics 项目、有明确 target / candidate、有 8-12 周决策压力、能做 expression / binding / developability，并可逐步接入肿瘤模型 reward。
  • 交付形态：可以是付费 Sprint、共研项目、标杆试点或带后续权益的合作，但必须服务 AI4AI 模块验收。
Step 4

项目回流驱动 AI4AI 增长

Sci 项目最大的价值不是单次交付，而是把真实证据、工具轨迹、专家分歧、wet reward、negative samples 和失败恢复路径沉淀回来。

  • 解决的问题：AI4AI 的成长需要真实 reward，而不是只靠人工写 prompt 或离线报告。
  • 回流产物：eval set、preference data、critique data、failure-recovery samples、tool-use traces 和下一轮 agent 改进任务。

04 · Work Package

第一阶段工作包

第一阶段不是先卖完整平台，也不是立刻做完整 biotech 管线；工作包分两层：内部先交付 AI4AI core v0，外部用 Sci 合作项目做逐步牵引。Target-to-Binder Sprint 只是 Sci 项目成熟后的最小可交付包装。

AI4AI core v0

  1. 先建设 agent runtime、project workspace、memory graph、tool registry、reward ledger、training data compiler 和 eval / replay harness。
  2. 验收标准是能组织复杂任务、留下轨迹、回放决策、导出评测样本和训练数据。
  3. 这是第一阶段资源申请的核心，不依赖某个 Sci 单子先成交。

05 · Delivery System

项目交付系统

这一层回答 Sci 项目如何变成 AI4AI 的真实训练场：从项目输入，到工作区推进，再到交付档案、实验数据包和 reward 回流，最后在授权边界内沉淀成下一次可复用的研发流程和训练数据。

AI4AI 验收问题

每个 Sci 项目开始前，先明确要验收哪个 AI4AI 模块：memory 是否能追溯证据，tool orchestration 是否可复现，ledger 是否能记录 reward，compiler 是否能产出训练样本。

Sci 项目输入

合作方给出 target、肿瘤疾病假设、候选分子、项目约束、成功标准和停止标准。第一步不是跑模型，而是确认是否适合做蛋白质设计和标准 wet reward。

交付生成

按用户场景导出候选物决策档案、实验与数据包、管理层摘要、内部试点评估材料或论文材料，让项目结果能被继续使用。

AI4AI 回流

每个项目结束后，在授权范围内留下模板、数据标准、失败案例、评审规则、实验反馈、复盘记录和训练样本，成为下一次 agent 迭代的基础。

06 · Technical Architecture

AI4AI 核心架构

这一层是衔远内部第一阶段真正要沉淀的能力：面向复杂专业任务的 agent 运行系统。Sci 项目只是第一批高价值场景，所有证据、工具调用、专家判断、reward 和可回放轨迹都要回到 AI4AI 核心架构里。

1
Scientific Memory Graph把文献、数据库、内部材料和项目状态整理成 claim-evidence-provenance 图谱，保留来源、版本和冲突证据
2
Scientific Tool Orchestration统一调度结构预测、序列设计、数据库 API、文献检索、内部模型和 wet-lab partner 接口，并约束输入输出格式
3
Review & Reward Ledger记录专家 rubric、候选排序、人工 override、QC flags、expression / binding / developability、肿瘤模型 reward 和分歧意见，让“好结果”可追踪、可比较
4
Execution Trace & Replay每次工具调用、候选变更、证据引用、审批 gate 和数据快照都留下轨迹，支持审计、复盘和复现实验路径
5
Training Data Compiler把内部生成轨迹、schema、rubric、匿名统计和授权数据编译为 SFT、preference、critique、failure recovery 和 RL-ready transition
6
Agent Runtime & Project Room提供项目隔离、权限审计、任务计划、审批 gate、预算控制、客户原始数据默认不训练和交付导出

技术特色

普通 RAG 只能回答问题，普通 CRO 只能交付实验，普通报告工具只能整理材料。我们的核心是把证据、工具、评审、实验反馈和执行轨迹放在同一套可复盘、可审计、可训练的 AI4AI 系统里；AGI4Sci 的价值是提供真实任务和真实 reward。

Memory证据图谱、项目状态和版本化科研记忆
Tool可组合、可约束、可复现的科学工具调用
Reward专家判断、质控标准、wet reward 和肿瘤模型 phenotype reward
Trace可回放轨迹和授权边界内的训练数据资产

07 · Growth Loop Example

AI4AI 增长闭环：Sci 项目如何反哺核心能力

这一节不再强调某个具体单子，而是说明 Sci 项目进入之后，如何从真实任务变成 AI4AI 的评测、训练和能力增长来源。

一次项目怎么跑

从 AI4AI 验收问题到 Sci 反馈回流，每一步都留下证据、版本和复盘材料。

1先定义本轮要验证的 AI4AI 能力：memory、tool orchestration、reward ledger、compiler 或 agent runtime
2外部伙伴输入肿瘤 target、疾病场景、候选物或项目约束，明确成功标准和停止标准
3系统整理文献、数据库、竞品、结构信息和项目材料，形成可追溯证据图
4AI4AI tool orchestration 调度模型和工具，生成或筛选 binder、antibody、nanobody 或 protein candidate
5外部专家评审候选排序、机制假设、成药性风险和实验可行性，评审过程进入 rubric
6Top candidates 进入 expression、binding、developability，后续接入 organoid / organ-chip phenotype reward pilot
7输出项目建议，同时把失败案例、实验反馈、专家判断和工具轨迹编译成 eval / training data
8用这批真实数据推动下一轮 AI4AI agent、tool policy、memory schema 和 reward model 改进

为什么用 Sci 牵引 AI4AI

Sci 不是一开始的全部目标，但它特别适合暴露 AI4AI 的真实短板。

1任务复杂：Sci 同时需要文献、结构、序列、实验、专家判断和项目决策，比普通 benchmark 更能检验 agent
2反馈真实：expression、binding、developability 和 organ-chip reward 能形成比文本评价更硬的学习信号
3周期可控：8-12 周能形成候选排序、标准 wet reward 和下一轮实验判断，适合第一批牵引项目
4边界清楚：Target-to-Binder 是可控前端工作包，ADC / 双抗 / TCE 是后续治疗路径，不会把范围撑散
5长期可选：当闭环稳定后，再讨论 AI-native biotech、共研权益或资产分成，而不是第一天就承诺全栈 biotech

08 · Validation Plan

18 个月验证计划

18 个月不是一上来冲完整 Sci 业务，而是让 AI4AI core 从内部可运行，逐步走到真实 Sci 项目牵引，再走到跨项目复用。每一阶段都要明确：AI4AI 核心模块验收是什么，Sci 合作方推进到哪里，真实反馈如何回流。

0-3 个月

AI4AI core v0 与 Sci 伙伴池

第一阶段先把 AI4AI 可运行底座搭起来；Sci 侧不急着拍死 target，而是确定合作方名单、项目选择标准、实验接口和数据边界。

  • AI4AI 通过标准：agent runtime、project workspace、memory schema、tool registry、ledger schema、compiler schema 和 eval / replay harness 可运行。
  • Sci 通过标准：确定候选生物公司、target / disease hypothesis 选择标准、成功标准、停止标准、专家 rubric、实验伙伴 shortlist 和数据字段。
  • Agent 平台：项目空间、租户隔离、权限审计、任务模板、human approval gate 和运行日志上线。
  • Memory Graph：claim-evidence-provenance schema、target / candidate / assay / entity model、文献 / 专利 / 数据库 ingest v0。
  • Tool Orchestration：文献检索、结构 / 序列工具、蛋白候选生成、数据库 API 的 tool registry 和标准 I/O。
  • Review & Reward Ledger：专家 rubric、候选评分、人工 override、QC flags、wet reward 和 tumor phenotype reward 字段定义。
  • Training Data Compiler：trace schema、脱敏 / 授权边界、SFT / preference / critique / failure-recovery 数据格式定义。
3-6 个月

AI4AI dry-run 与 Sci 项目选择

先用公开数据和 partner-safe 任务做 dry-run，验证 AI4AI 模块能否闭环；同时从合作方项目池里选出第一个真实 Sci 牵引项目。

  • AI4AI 通过标准：从任务计划、证据整理、候选生成、专家评审到训练样本导出，全流程可回放、可评测、可复现。
  • Sci 通过标准：第一个外部项目完成 problem framing，明确 target / candidate、实验边界、授权边界和 wet reward 可行性。
  • Agent 平台：能自动生成任务计划、预算 / 时间记录、失败重试、审批节点和项目时间线。
  • Memory Graph：每条关键结论能追溯来源、冲突证据、版本和引用。
  • Tool Orchestration：候选生成、筛选、去重和风险标注可复现，工具输入输出可回放。
  • Review & Reward Ledger：专家评分差异、反驳理由和实验优先级结构化入库。
  • Training Data Compiler：dry-run 轨迹可导出为训练 / 评测样本，包含成功、失败、人工修正和恢复路径。
6-12 个月

真实 Sci 项目牵引 AI4AI 增长

第一个真实 Sci 项目进入执行：候选生成、专家评审和小规模 wet reward 都要服务 AI4AI 回流，而不是只交付一次报告。

  • AI4AI 通过标准：真实项目产生的 raw data、negative samples、专家分歧、失败原因和工具轨迹可进入 eval / training data pipeline。
  • Sci 通过标准：Top candidates 完成 expression、binding、developability，小规模 organoid / organ-chip phenotype reward pilot 视项目成熟度接入。
  • Agent 平台：wet-lab SOP、样本表、状态、raw data、QC flags 接入项目流。
  • Memory Graph：实验结果回写到 target、candidate、evidence、assay 和 tumor model 节点，形成版本化项目记忆。
  • Tool Orchestration：能根据 wet feedback 触发候选重排、补实验建议或下一轮候选生成。
  • Review & Reward Ledger：raw data、negative samples、失败原因、专家复评、wet reward 和 tumor phenotype reward 全量可查询。
  • Training Data Compiler：形成首批 SFT / preference / critique / failure-recovery / eval set，不含客户未授权原始数据。
12-18 个月

跨项目复用与长期选项判断

接入第二个 Sci 项目，判断同一套 AI4AI core 是否可复用，并让启动成本、交付周期和人工组织成本明显下降；长期再判断是否进入共研权益、里程碑、分成或 AI-native biotech。

  • AI4AI 通过标准：第二项目复用 runtime、memory schema、tool policy、ledger schema、compiler pipeline 和评测集，关键步骤无需重写。
  • Sci 通过标准：第二个外部项目能复用 target intake、protein design workflow、expert rubric、实验数据模板和决策档案结构。
  • Agent 平台：复用项目模板、权限模型、任务计划、审批 gate 和交付导出。
  • Memory Graph：复用 schema 和 ingest pipeline，新增 target 的初始化时间下降。
  • Tool Orchestration：复用工具链、评测 harness 和候选生成流程，关键步骤无需重写。
  • Review & Reward Ledger：复用 rubric / wet reward schema，能跨项目比较成功和失败模式。
  • Training Data Compiler：形成可持续训练 / 评测数据生产线，支持下一轮 agent 能力迭代和私有部署判断。
  • 经营判断：决定是否继续做付费 Sprint、垂直 project room、私有部署、共研权益或资产分成；前提是 AI4AI core 已经被真实项目证明能复用。
