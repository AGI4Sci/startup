01 · Goal

目标与边界

回答四件事：最终要建设什么、第一年验证什么、哪些资产会形成长期复利，以及第一阶段不做什么。

Company

长期目标

建设下一代科学研究基础设施：连接科学知识、计算工具、专家判断和实验反馈，支持 AI 在真实科研任务中持续改进。

  • AI 自进化目标：支撑 AI agent 多尺度递归改进：模型能力、科学记忆、agent harness（执行/记录/回放/评测框架）和科研实验环境共同进化。
  • 科学影响力目标：让 AI 参与真实科学发现：从文献理解、假设生成、实验设计到结果复盘，形成可复现、可审查、可发表的科研过程。
  • 产业影响力目标：把科研线索转化为产业方能判断、能行动的项目资产：靶点判断、候选设计、实验验证、对外合作、临床前申报或产品化决策。
Year 1

第一年验证

用 1 个可验证、可讲故事、不过度敏感的生命科学项目 + 1 个关键 ML 任务验证四件事：AI4AI 平台底座、AI 自进化方法、科学质量、产业价值。

  • AI4AI 平台验证：Scientific Memory Graph、Scientific Tool Orchestration、Review & Reward Ledger、Training Data Compiler 和 agent 平台本身完成 v0 验收，能支撑真实项目执行、回放和数据编译
  • AI 自进化验证：选择 1 个关键 ML 场景(例如长序列建模、蛋白结构预测、长文本 RL 算法)，在公开 benchmark 或可复现实验上达到 / 超过当前 SOTA
  • 科学验证：围绕 1 个公开证据较多、结构或抗体设计空间明确、IP 不过度敏感的 target，形成一套可复盘的靶点判断材料
  • 产业验证：为 1 个项目型 biotech 完成 Target-to-Binder Sprint，8-12 周内生成 50-300 条候选，Top 20-30 条进入表达、结合和成药性验证
Asset

复利资产

Agent 平台负责把项目跑起来，Memory 是 source of truth，Tool、Ledger 和 Compiler 分别负责工具调用、科学判断和训练自进化。

  • Agent Runtime & Project Room：项目空间、权限、任务计划、审批 gate、执行日志和交付导出
  • Scientific Memory Graph：文献、数据库、claim-evidence-provenance、项目状态、版本和冲突证据
  • Scientific Tool Orchestration：深度学习模型、计算工具、数据库 API、MCP 服务、wet-lab 接口
  • Review & Reward Ledger：expert taste、rubric、wet reward、human override、失败原因和 QC flags
  • Training Data Compiler：基于内部轨迹、schema、rubric、匿名统计和授权数据形成 SFT、preference、critique、failure recovery 和 agent self-evolve dataset
Boundary

第一阶段边界

第一阶段不是卖通用科研助手或总部级平台订阅，而是用 Target-to-Binder Sprint 服务有明确决策窗口的 discovery team。

  • 客户与伙伴边界：首批直接付费客户锁定 ADC / 双抗 / TCE / biologics 项目型 biotech；大药企只做单项目标杆试点，不作为第一收入主线；科研机构主要服务学术影响力和高质量项目源；实验验证伙伴多是我们付费采购或重点共建的对象。
  • 产品边界：Target-to-Binder Sprint 是第一单；Project Room 是交付载体，也是 Science Agent OS 的产品雏形；完成 2-3 个项目后再评估平台订阅、私有部署或企业 workflow。
  • 市场边界：第一阶段只服务有明确靶点或候选分子的生物药早研项目，帮助客户在 8-12 周内决定继续投入、停止、对外 BD，或进入下一轮实验；不覆盖小分子全流程、临床开发、常规 CRO 外包和通用科研工具。
  • 数据边界：客户原始数据、项目材料、序列和实验结果默认不用于训练；schema、rubric、workflow、匿名统计、授权数据和内部生成轨迹可用于平台改进。

03 · First Projects

第一批项目题目

第一优先级不是提前锁死某家公司的未公开项目，而是把 ADC / 双抗 / TCE / biologics 早研团队的公开管线和项目压力，翻译成 4 类可以约谈、可以收费、也能验证平台的 Sprint 题目。

Project A

ADC 靶点 / 候选差异化

适配 DualityBio、MediLink、LaNova、Mabwell、Lepu 等 ADC 管线密集团队。

  • 具体问题：同一靶点、适应症或 payload 方向竞争拥挤，团队需要在 BD、融资或下一轮实验前判断 target / candidate 是否足够差异化。
  • 第一阶段交付：竞品与靶点表达证据、linker-payload / internalization / bystander / developability 风险、Top candidates 和下一轮 expression / binding / developability 验证建议。
Project B

双抗 / TCE format 选择

适配 EpimAb、Alphamab、Harbour BioMed、Keymed、LaNova 等双抗、多抗和 TCE 团队。

  • 具体问题：target pair、亲和力窗口、format、T cell activation、cytokine risk 和 off-tumor 风险没有统一判断口径。
  • 第一阶段交付：机制假设、format review、专家 rubric、候选排序和 binding / activation / developability 实验优先级。
Project C

BD 前证据缺口补齐

适配 DualityBio、LaNova、OnCusp、Elpiscience、ImmuneOnco 等有出海、授权或融资叙事的项目。

  • 具体问题：海外 partner、投资人或内部评审会追问差异化、可开发性、实验缺口和反方证据，现有材料往往散在文献、PPT 和专家意见里。
  • 第一阶段交付：可复盘的项目判断材料、竞品地图、关键反驳、补实验清单和 partner-ready 摘要。
Project D

backup / next-gen 候选排序

适配 Mabwell、Harbour BioMed、Keymed、MediLink、Adagene 等已有抗体、ADC 或工程平台的团队。

  • 具体问题：lead candidate 之外是否要做 backup、换 epitope、改 format、换 payload、进入新适应症，常常缺少统一证据和失败记录。
  • 第一阶段交付：候选排序、developability 标签、失败风险记录、下一轮验证样本表和可回放的决策链。

04 · Product Offer

产品形态

第一年只打一条 Target-to-Binder Sprint 闭环。衔远交付 AGI4Sci 平台能力和项目操作系统，外部伙伴牵引具体靶点、专家评审和实验验证；客户拿走候选生成、证据整理、小规模 wet validation 和最终决策档案。

项目工作区

  1. 客户和我们一起推进项目的在线空间。
  2. 统一放置靶点证据、候选分子、实验进展、专家意见和关键决策。
  3. 方便查看进度、追溯依据、导出汇报和归档材料。

衔远 AGI4Sci 平台能力

  1. 建设 project room、agent workflow、工具编排、证据记忆、执行回放和评测 harness。
  2. 把每次候选生成、专家反馈、实验结果和项目决策转成可审计轨迹。
  3. 沉淀 schema、rubric、评测 harness 和授权数据资产，服务下一轮 AGI4Sci 平台迭代。

外部伙伴案例牵引

  1. 外部伙伴负责具体 target 背景、疾病假设、科学可行性和专家评审。

  3. 衔远不依赖自建完整科学项目来源，而是把伙伴案例和实验能力接入统一 AGI4Sci 项目系统。

05 · Delivery System

项目交付系统

这一层回答 Target-to-Binder Sprint 如何被生产出来：从项目输入，到工作区推进，再到交付档案和实验数据包，最后在授权边界内沉淀成下一次可复用的流程。

项目输入

客户给出 target、疾病假设、候选分子、项目约束、成功标准和停止标准。第一步不是跑模型，而是确认 target 是否公开证据充分、IP 不过度敏感，并把数据默认不训练、授权复用边界和判断口径说清楚。

项目工作区

所有证据、候选、实验、专家意见和决策记录进入同一个 Web 工作区。客户看到的是项目进展和判断依据，不需要理解底层技术模块。

交付生成

按用户场景导出候选物决策档案、论文材料、实验与数据包、管理层摘要或内部试点评估材料，让项目结果能被继续使用。

项目沉淀

每个项目结束后，在授权范围内留下模板、数据标准、失败案例、评审规则、实验反馈和复盘记录，成为下一次项目更快、更准的基础。

06 · Technical Architecture

平台技术架构

这一层是衔远内部要沉淀的 AI4AI / AGI4Sci 平台能力：底层复用 AI4AI 方法，产品形态面向科学任务，把 agent 平台、科研记忆、工具编排、专家判断、实验反馈和可回放轨迹组织成持续改进的系统。

1
Scientific Memory Graph把文献、数据库、内部材料和项目状态整理成 claim-evidence-provenance 图谱，保留来源、版本和冲突证据
2
Scientific Tool Orchestration统一调度结构预测、序列设计、数据库 API、文献检索、内部模型和 wet-lab partner 接口，并约束输入输出格式
3
Review & Reward Ledger记录专家 rubric、候选排序、人工 override、QC flags、expression / binding / developability 结果和分歧意见，让“好结果”可追踪、可比较
4
Execution Trace & Replay每次工具调用、候选变更、证据引用、审批 gate 和数据快照都留下轨迹，支持审计、复盘和复现实验路径
5
Training Data Compiler把内部生成轨迹、schema、rubric、匿名统计和授权数据编译为 SFT、preference、critique、failure recovery 和 RL-ready transition
6
Agent Runtime & Project Room提供项目隔离、权限审计、任务计划、审批 gate、预算控制、客户原始数据默认不训练和交付导出

技术特色

普通 RAG 只能回答问题，普通 CRO 只能交付实验，普通报告工具只能整理材料。我们的核心是把证据、工具、评审、实验反馈和执行轨迹放在同一套可复盘、可审计、可在授权边界内持续改进的科研运行系统里。

Memory证据图谱、项目状态和版本化科研记忆
Tool可组合、可约束、可复现的科学工具调用
Reward专家判断、质控标准和首阶段 wet reward 记录
Trace可回放轨迹和授权边界内的训练数据资产

07 · Vertical Loop Example

第一条闭环：Target-to-Binder Sprint

这不是要衔远自建所有科学案例来源，而是用外部伙伴的真实项目牵引 AGI4Sci 平台能力：周期短、交付清楚，expression / binding / developability 的 wet reward 明确。

一次项目怎么跑

从客户输入到决策交付，每一步都留下证据、版本和复盘材料。

1客户和外部伙伴输入 target、疾病假设、候选物或项目约束，明确成功标准和停止标准
2衔远系统整理文献、数据库、竞品、结构信息和项目材料，形成可追溯靶点证据图
3衔远 AGI4Sci 平台调度模型和工具，生成或筛选 binder、antibody、nanobody 或 format options，并给出风险标签
4外部专家评审候选排序、机制假设、成药性风险和实验可行性，评审过程进入 rubric
5Top 20-30 candidates 进入外部 wet-lab partner 的 expression、binding、developability 小规模验证
6输出继续、停止、BD、补实验或下一轮优化建议
7把失败案例、实验反馈、专家判断和工具轨迹回流到衔远 AGI4Sci 平台

为什么先从这里开始

它同时具备商业清晰度、实验反馈和技术复用价值。

1输入清楚：客户通常已有 target、疾病场景或候选分子
2周期可控：8-12 周能形成候选排序和小规模验证结果
3交付明确：客户能用结果做 continue、kill、BD 或下一轮实验决策
4反馈真实：expression、binding、developability 给出可复盘的 wet reward，不需要衔远先自建实验组织
5可扩展：同一 AGI4Sci 平台可延展到 binder optimization、BD / IND / 论文材料；phenotype reward 单独作为第二 pilot

08 · Validation Plan

18 个月验证计划

每个阶段同时验收两条线：外部伙伴能否带来真实案例和 wet reward；衔远 AI4AI 平台的 agent runtime、Memory Graph、Tool Orchestration、Review & Reward Ledger、Training Data Compiler 是否能在真实项目中完成可验收的搭建、运行和复用。

0-3 个月

AI4AI 平台 v0 与项目定义

确定 1 个公开证据较多、结构或抗体设计空间明确、IP 不过度敏感的 target；外部伙伴给出案例牵引、科学输入和实验边界，衔远先把可运行底座搭起来。

  • 外部项目：target、成功标准、停止标准、专家 rubric、实验伙伴 shortlist 和数据字段确认。
  • Agent 平台：项目空间、租户隔离、权限审计、任务模板、human approval gate 和运行日志上线。
  • Memory Graph：claim-evidence-provenance schema、target / candidate / assay / entity model、文献 / 专利 / 数据库 ingest v0。
  • Tool Orchestration：文献检索、结构 / 序列工具、候选生成、数据库 API 的 tool registry 和标准 I/O。
  • Review & Reward Ledger：专家 rubric、候选评分、人工 override、QC flags 和 wet reward 字段定义。
  • Training Data Compiler：trace schema、脱敏 / 授权边界、SFT / preference / critique / failure-recovery 数据格式定义。
3-6 个月

dry-run 与模块验收

围绕选定 target 完成 dry-run：证据整理、候选生成、专家评审和候选排序；重点不只看候选，更看平台模块是否能复盘、回放和导出训练样本。

  • 外部项目：生成 50-300 条候选，Top 20-30 条完成外部专家评审和实验优先级排序。
  • Agent 平台：能自动生成任务计划、预算 / 时间记录、失败重试、审批节点和项目时间线。
  • Memory Graph：每条关键结论能追溯来源、冲突证据、版本和引用。
  • Tool Orchestration：50-300 条候选的生成、筛选、去重和风险标注可复现，工具输入输出可回放。
  • Review & Reward Ledger：Top 20-30 条候选完成专家评审，评分差异、反驳理由和实验优先级结构化入库。
  • Training Data Compiler：dry-run 轨迹可导出为训练 / 评测样本，包含成功、失败、人工修正和恢复路径。
6-12 个月

wet reward 与数据闭环

Top 20-30 candidates 进入外部伙伴的表达、结合和成药性验证；实验回流后同时检查 AGI4Sci 项目结果和 AI4AI 数据飞轮是否成立。

  • 外部项目：交付候选物决策档案，给出继续、停止、BD 或补实验建议。
  • Agent 平台：wet-lab SOP、样本表、状态、raw data、QC flags 接入项目流。
  • Memory Graph：实验结果回写到 target、candidate、evidence 和 assay 节点，形成版本化项目记忆。
  • Tool Orchestration：能根据 wet feedback 触发候选重排、补实验建议或下一轮候选生成。
  • Review & Reward Ledger：raw data、negative samples、失败原因、专家复评和 wet reward 全量可查询。
  • Training Data Compiler：形成首批 SFT / preference / critique / failure-recovery / eval set，不含客户未授权原始数据。
  • AI4AI 方法：关键 ML benchmark 达到强基线，并沉淀 harness、评测流程和失败复盘模板。
12-18 个月

第二项目复用与平台化判断

不以自营具体科学案例来源为目标，而是判断同一套 AGI4Sci 平台能否接入第二个外部伙伴项目，并让启动成本、交付周期和人工组织成本明显下降。

  • 外部项目：第二个外部项目能复用 target intake、expert rubric、实验数据模板和决策档案结构。
  • Agent 平台：复用项目模板、权限模型、任务计划、审批 gate 和交付导出。
  • Memory Graph：复用 schema 和 ingest pipeline，新增 target 的初始化时间下降。
  • Tool Orchestration：复用工具链、评测 harness 和候选生成流程，关键步骤无需重写。
  • Review & Reward Ledger：复用 rubric / wet reward schema，能跨项目比较成功和失败模式。
  • Training Data Compiler：形成可持续训练 / 评测数据生产线，支持下一轮 agent 能力迭代和私有部署判断。
  • 经营判断：决定是否继续做垂直 project room、私有部署或企业 workflow，而不是扩张重资产实验能力。
